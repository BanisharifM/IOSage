## 3. Domain Shift Analysis: Four Contributing Factors

### Factor 1: Prior Inversion
- **Polaris training data**: 54% of traces are labeled "healthy" (no bottleneck). Bottleneck dimensions have low prevalence: interface_choice 1.2%, file_strategy 1.4%, parallelism_efficiency 1.6%.
- **TraceBench**: 0% healthy traces (all 28 traces exhibit at least one issue). interface_choice appears in 75% of traces, file_strategy in 50%, parallelism_efficiency in 64%.
- **Impact**: The model's prior strongly favors "healthy" predictions, producing empty bottleneck sets for 24/28 TraceBench traces at the default threshold.

### Factor 2: Label Methodology Mismatch
- **Polaris labels**: Generated by Drishti heuristic rules applied to Darshan counters. Drishti uses fixed thresholds (e.g., sequential_ratio > 0.5 implies sequential access pattern).
- **TraceBench labels**: Assigned by human experts examining application source code and I/O behavior. Labels like "NC-R" (non-collective read) and "MPNM" (MPI non-MPI) reflect code-level design decisions, not just counter-based thresholds.
- **Impact**: The same underlying I/O behavior may be labeled differently. For example, a trace with small random reads might be labeled "SML-R" by TraceBench (based on transfer size) but not flagged by Drishti if the total read time is below its threshold.

### Factor 3: System and Workload Shift
- **Polaris**: ALCF production system with Lustre file system (160 OSTs, 650 GiB/s peak). Traces represent real scientific applications.
- **TraceBench**: Mix of IO500 benchmarks, h5bench, and real applications from various HPC systems. Different storage hardware, file system configurations, and I/O stack versions.
- **Impact**: Feature distributions shift. For example, absolute throughput values, file counts, and timing patterns are system-dependent. While our normalization (log1p + RobustScaler) mitigates some of this, the model learned Polaris-specific decision boundaries.

### Factor 4: Taxonomy Mapping Ambiguity
- **TraceBench** defines 13 fine-grained issue labels (SML-R, SML-W, MSL-R, MSL-W, RDA-R, RDA-W, NC-R, NC-W, MPNM, LLL-R, LLL-W, SHF, HMD, SLIM, RLIM, NOL).
- **Our taxonomy** defines 8 dimensions (access_granularity, metadata_intensity, parallelism_efficiency, access_pattern, interface_choice, file_strategy, throughput_utilization, healthy).
- **Impact**: The many-to-one mapping introduces noise. For instance, TraceBench's "SLIM" (serialization at a single process limiting throughput) maps to parallelism_efficiency, but our parallelism_efficiency dimension was trained primarily on rank-level variance patterns from Drishti, which detects a different manifestation of the same high-level issue.

## 4. Comparison with Other Approaches on TraceBench

| Approach | Metric | Score | Method |
|---|---|---|---|
| **IOAgent** (IPDPS'25) | LLM-ranked accuracy | 0.641 | GPT-4 + RAG over darshan-parser text output |
| **IONavigator (ours)** | Micro-F1 (default) | 0.103 | XGBoost on 186 Darshan features, threshold=0.5 |
| **IONavigator (ours)** | Micro-F1 (LOO-CV) | 0.163 | Same model, recalibrated thresholds |
| **IONavigator (ours)** | Mean AUC-ROC | 0.591 | Probability ranking quality (no threshold) |
| **IONavigator on Polaris GT** | Micro-F1 | 0.923 | Same model, in-distribution test set (436 samples) |
| **IONavigator + closed-loop** | Geo-mean speedup | 7.2x | ML detection + LLM recommendation + re-run |

Note on metric comparability:
- IOAgent's "LLM-ranked accuracy" counts a trace as correct if the top-ranked LLM-identified issue matches any ground-truth label. This is a lenient single-label metric.
- Our Micro-F1 requires predicting the correct *set* of dimensions per trace (multi-label). A trace with 4 true issues where we predict 1 correctly scores ~0.4 recall but 1.0 precision for that prediction.
- The metrics are not directly comparable, but both measure the same fundamental task: identifying I/O bottleneck categories from Darshan traces.

## 6. Paper-Ready Text: Cross-Dataset Generalization Subsection

### Subsection: Cross-Dataset Generalization

To evaluate generalization beyond our Polaris-trained model, we test on TraceBench, an independent benchmark suite of 35 Darshan traces with expert-assigned I/O issue labels from multiple HPC systems. After mapping TraceBench's 16 fine-grained labels to our 8-dimension taxonomy, we parse 28 traces (7 fail due to incompatible Darshan log format).

At the default threshold (0.5), our model achieves Micro-F1 = 0.103 on TraceBench, compared to 0.923 on in-distribution Polaris test data. A probability diagnostic reveals that while absolute predicted probabilities are severely miscalibrated (mean P(positive) for true positives < 0.01 on TraceBench vs. > 0.9 on Polaris), the relative ranking preserves partial signal: mean AUC-ROC across dimensions with both positive and negative examples is 0.591, with metadata_intensity achieving 0.860 and access_pattern 0.787.

LOO-CV threshold recalibration improves TraceBench Micro-F1 to 0.163 (1.59x improvement), but degrades Polaris performance from 0.923 to 0.848. This confirms the performance gap stems from genuine domain shift---not threshold miscalibration---driven by four factors: (1) prior inversion (0% healthy in TraceBench vs. 54% in training data), (2) label methodology mismatch (Drishti heuristics vs. expert source-code analysis), (3) system-level feature distribution shift, and (4) taxonomy mapping ambiguity.

This result is consistent with findings from OpenRCA (ICLR 2025), which showed that the best LLM-only approach achieves 11.34% accuracy on root cause analysis when label definitions change between datasets. It motivates our hybrid architecture: the ML component provides high-accuracy detection within its training domain (F1 = 0.923), while the LLM component enables adaptation to new contexts through reasoning over SHAP explanations and benchmark knowledge---a capability we demonstrate through 7.2x geometric mean speedup in closed-loop validation.

## 8. Summary and Verdict

| Question | Answer |
|---|---|
| Does the model have signal on TraceBench? | Partial (AUC-ROC = 0.591), mainly in metadata_intensity and access_pattern |
| Does threshold recalibration help on TraceBench? | Modestly (0.103 -> 0.163, +58%) |
| Does recalibration hurt Polaris performance? | **Yes, significantly** (0.923 -> 0.848, -8.1%) |
| Should we use recalibrated thresholds in production? | **No**. The cost on in-distribution data outweighs the gain on out-of-distribution data. |
| What should we report in the paper? | Report both results as evidence of domain shift. Use the default threshold for all primary metrics. Present the diagnostic as an honest analysis of generalization limitations. |
| Does this hurt our paper? | No -- it strengthens it. Honest reporting of generalization challenges, combined with the hybrid ML+LLM architecture that addresses them, is more compelling than hiding the gap. |
